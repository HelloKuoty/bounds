# 《界 / Bounds》· 评审与开发组人物档案

> 用于模拟一个**产品迭代闭环**:
> 玩家评审委员会出意见 → 用户研究员去重归因 → 产品经理排优先级 → 开发组执行 → QA 验证 → 下一轮同一批人复评。
>
> 这些是**虚拟角色**(供 subagent 扮演),不是真人。每个角色都刻意带不同立场与盲区,让意见互相打架——这正是价值。

## C. 运作方式(评审循环)

1. **评审**:每个玩家角色(①–⑬)基于游戏**真实状态**给出 意见 + 评分(上手/深度/主题/想不想推荐)+ 一句毒舌总结。
2. **汇总**:明察(用户研究)去重、归因、量化,标出共识与分歧。
3. **定位与排期**:沈牧(PM)先逼出目标用户定位,再把问题排成「必修 / 想做 / 不做」路线图。
4. **执行**:江决/乔禾/白川/沉吟/言叙/韩磬 各领其责;程衡守保真,找茬(QA)配回归测试。
5. **复评**:同一批玩家角色复玩,看必修项是否真的解决——形成闭环。

> 评分量表(1–5):**上手** / **深度** / **主题(寓言是否成立)** / **推荐意愿(NPS 式)**。
