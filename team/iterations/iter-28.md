- 基线绿(tests 105 / smoke OK)。承接转向:重玩核心——随机布局 + "保证有解"。

## 做了什么
- ✅ **种子化生成器 `TerritoryGen.make(seed) -> TerritoryData`**:
  - 产出 1–2 个**名冲突**(同 glyph、两 meaning)+ 时而一个**"异名同心"簇**(同一 essence、不同名,须成束);表面(glyph/label/essence)从**在世词池**随机抽,glyph 全局不重(只有有意的冲突对共字)→ 无杂散失稳。
  - **构造即可解**:每冲突一记画界、簇一记成束;`concord_target = 麻烦数`,`insight = 麻烦数 + 1`(略紧)。`TerritoryData.from_dict(dict)` 直接构造,无需改数据层。
- ✅ **接入 run**:`GameState.run_seed`(`start_new_run` 时 `randi()`,**每 run 不同**;存档持久 → **retry/reload 稳定**);`main._open_territory` 对**不在 JSON 数据库的 id** 走生成器;`RunManager.PLAN` 分叉层加 **「试炼之地」节点**(`terr:"trial"`,`seed = hash(node_id) ^ run_seed`)。
- ✅ **可解性保证测试** `tests/test_territory_gen.gd`:**60 个种子全部**用通用贪心解算器复跑通关;同种子复现(retry 公平)、不同种子相异;**生成文本逐串扫禁词**(铁律对生成内容同样成立)。新 `class_name` 先 `--import`。
- → **108 绿**;smoke 通关(试炼在分叉 index 3、烟雾走 options[0] 不经过,但引擎由专测覆盖);LOAD 0;重导出网页。

## 为什么这治"内容太少"
